# Product Decision

最終更新: 2026-04-11

## 1. 何を決めたか

- 現行プロジェクトは閉鎖する
- 新プロジェクトとしてクリーンに作り直す
- 本命 UX は `overlay` ではなく `memo 1 window`
- 対象プラットフォームは当面 `macOS` のみ

## 2. なぜ作り直すのか

- 問題の中心がバグ修正ではなく、表示モデルの前提変更だから
- `single overlay` 前提の上に `multi window` を継ぎ足すと責務が混ざるから
- 試行錯誤の分岐が増え、今後の保守性が下がるから
- 今回の価値は web UI よりも `macOS の自然な window 挙動` にあるから

## 3. 新プロジェクトのプロダクト定義

- 用途:
  - 思考断片を即座に退避する macOS メモアプリ
- コア体験:
  - ショートカットで出す
  - すぐ書く
  - すぐ元作業に戻る
  - 1 click でまた触る
- 管理思想:
  - まず退避
  - 整理は後から

## 4. MVP で最優先にするもの

- 起動から入力開始までの速さ
- 1 memo window の自然さ
- 1 click での再編集
- pin/unpin の直感性
- close/reopen の安心感

## 5. MVP で優先しないもの

- canvas 的な一括空間操作
- 複数 memo の同時移動
- 9 枚 session や 15 memo 展開を前提にした体験
- cross-platform

## 6. 非ゴール

- 既存コードの延命
- overlay / through の再挑戦
- 初期段階での高度な複数選択 UX
- 同期、共有、AI 整理

## 7. 既存データの扱い

- 旧 `Sticky` の DB や Trash は、初期 native MVP では自動移行しない
- 旧プロジェクトは legacy として残し、必要時は旧アプリ側で参照する
- 新 native 版は新規 DB で開始する
- 旧 DB 読み取りインポートは、native MVP 安定後の別フェーズで検討する

理由:

- 初期段階で移行機能まで入れると、window UX の検証と実装がブレる
- まずは新アプリ単体で `自然に書ける / 失わない` を成立させる方が優先順位として正しい
