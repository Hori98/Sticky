# Human Checklist

最終更新: 2026-04-11

この文書は、新 native プロジェクトへ移行するにあたり、人間側で決めること・準備すること・各フェーズで確認することを整理したチェックリストである。

## 1. 最初に決めること

- 新プロジェクト名
- bundle id
- GitHub repo 名
- macOS minimum version
- App Store 配布を将来考えるか
- 旧 `Sticky` をどの時点で完全停止するか
- 旧データ import を本当に初期スコープ外にするか

## 2. 先に準備すること

- Xcode が新規 native プロジェクトを作れる状態か確認する
- Apple ID / Signing の要否を確認する
- 実機で動作確認する Mac を決める
- 新 repo の作成先を決める
- 新 repo の default branch 名を決める

## 3. AI に最初に渡すべき入力

- app 名
- bundle id
- minimum macOS version
- menu bar app にするか dock app にするかの初期方針
- アイコンの方向性
- window の見た目の許容範囲
  - 角丸
  - 影
  - タイトルバー非表示
  - リサイズ UI

## 4. 人間が確認するべき UX

- `Cmd+Option+Enter` で本当に即書けるか
- 1 click で再編集が自然か
- pin の意味が直感的か
- close が不安なく使えるか
- memo window が仕事の邪魔にならないか
- menu bar 導線が自然か

## 5. フェーズごとの人間タスク

### Phase 0

- 仕様の優先順位を最終確認する
- 非ゴールを承認する
- 技術スタックを最終承認する

### Phase 1

- window の見た目と操作感を実機確認する
- drag / resize / pin / close の違和感を言語化する
- `1 memo = 1 window` で本当に行くか最終判断する

### Phase 2

- close/reopen と autosave の安心感を確認する
- 「失わない」が成立しているか確認する

### Phase 3

- Home / Trash / Settings の情報設計を確認する
- 後から回収する流れに無理がないか確認する

### Phase 4

- session を論理グループとして残す価値があるか確認する
- session 関連操作をどこまで入れるか判断する

### Phase 5

- 日常使用して違和感が残る箇所を洗い出す
- リリース前に削るものと残すものを判断する

## 6. 人間がやらなくていいこと

- 実装詳細の設計
- ファイル分割の判断
- コードレベルの技術調査
- タスクの細分化

これらは基本的に AI に任せてよい。

## 7. 人間がやると効果が高いこと

- 体感の良し悪しを短く具体的に伝える
- 迷いがある仕様を早めに二択で決める
- 「これは気持ちいい / これはキモい」を言語化する
- 実際の利用フローを例で伝える

## 8. 運用ルール

- `migration/` 配下を新プロジェクトの上位方針として扱う
- フェーズ詳細は別ファイルで追加する
- 実装開始後も、判断変更はまず文書に反映する
- 旧 `Sticky` は legacy として扱い、原則として拡張しない
