## 4. 問題一覧（Issue List）

### U-01: 疎通確認用 UI のままで、実メモ操作が存在しない
- 現状は接続確認用の付箋が表示されるだけで、sticky の本来操作を検証できない

### S-01: 選択・編集の状態遷移がコード上に未実装
- 文書上では `memo_selected -> editing` が定義されているが、React state として未実装

### U-02: 1枚セッションの可視表現が未実装
- 現在は `overlay probe` 表示であり、セッションやメモの概念が UI 上に存在しない

### U-03: クリック判定と drag threshold の基礎が未実装
- シングルクリック選択とダブルクリック編集を成立させるには、同一の pointer/mouse イベント列の中で `click` と `drag` を切り分ける必要がある

### U-04: メモ移動・リサイズの最小実装が未着手
- デフォルトサイズ、最小/最大値は決まっているが、UI 操作に落ちていない

### D-01: 明示保存の最小導線が未実装
- `Cmd + S` と `Cmd + Enter` の UI 上の効果確認ができない
- ただしこのフェーズでは、永続DB保存ではなくメモリ上の保存状態でもよい

### K-01: プローブ挙動と本実装の境界が曖昧
- 疎通確認用ショートカットと本番機能の責務分離が未整理

### K-02: 閉じた後の再表示フローが未定義
- `Cmd + Enter` で閉じた後に、Tauri からの再表示イベントを受けて UI を復元する流れが未整理

## 5. フェーズ内サブフェーズ
本フェーズは、主目的を維持したまま以下の小段階に分ける。

### Phase 2-1: メモUIの骨格置換
対応Issue:
- U-01
- U-02
- K-01

目的:
- `overlay probe` をやめ、1枚セッション前提の `MemoCard` 表現に置き換える

対象:
- `app/src/App.tsx`
- `app/src/App.css`
- 必要なら `app/src/index.css`

成果:
- sticky らしい見た目の単一メモカード
- 疎通確認ラベルではなく、実メモUIに近い骨格

Gate:
- 起動時に最小メモUIが表示される
- 前面レイヤーとしての挙動を壊していない
- 疎通確認用の状態表示が不要なら削除、必要なら最小化されている

### Phase 2-2: 選択・編集状態の実装
対応Issue:
- S-01
- U-01
- U-03

目的:
- `idle / memo_selected / editing` の最小状態遷移を React state で成立させる
- 併せて `drag threshold = 6px` の判定ロジックまで先に実装し、クリックとドラッグの分岐を一本化する

対象:
- `app/src/App.tsx`
- `app/src/App.css`

成果:
- シングルクリックで選択
- ダブルクリックまたは `Enter` で編集開始
- `Esc` で編集終了
- `mousedown -> mousemove` の中で `6px` 超過時にドラッグ開始候補として扱う
- `6px` 未満ならクリック確定として扱う
- IME composition 中は Enter による状態遷移を抑制する

Gate:
- 選択リング表示が切り替わる
- 入力可能/不可能が状態に応じて切り替わる
- `drag threshold = 6px` が実装され、クリックとドラッグの競合が起きない
- IME composition 中に Enter 起因の状態遷移が発火しない

### Phase 2-3: メモ移動・リサイズの最小実装
対応Issue:
- U-04

目的:
- Phase 2-2 で用意した `drag threshold` 判定の上に、1枚メモのドラッグ移動と角ドラッグリサイズの実処理を追加する

対象:
- `app/src/App.tsx`
- `app/src/App.css`
- 必要なら `app/src/lib` 配下に1ファイル追加

成果:
- ドラッグ移動
- 四隅ハンドル表示
- サイズ制約の適用

Gate:
- デフォルトサイズ `320x240` で表示される
- 最小 `240x180` / 最大 `520x420` が守られる
- メモをドラッグで任意の位置に移動できる
- 角ドラッグでメモをリサイズできる
- リサイズ中も選択リングが正しく表示される
- ドラッグ終了後に `memo_selected` 状態が維持される

### Phase 2-4: 明示保存と閉じるの最小実装
対応Issue:
- D-01
- K-01
- K-02

目的:
- `Cmd + S` と `Cmd + Enter` の意味を最小実装で確立する

対象:
- `app/src/App.tsx`
- 必要なら `app/src/lib` 配下1ファイル

成果:
- `Cmd + S`: 現在本文を更新保存
- `Cmd + Enter`: 更新保存して閉じる
- 閉じた後の表示状態切替
- Tauri からの再表示イベント受信時に、閉じたメモを再表示できる

Gate:
- `Cmd + S` 実行後、`isDirty = false` を確認できる
- `Cmd + Enter` で非表示化できる
- `Cmd + Enter` 後にグローバルホットキーで再表示できる
- 疎通確認ショートカットとの衝突がない

## 7. 設計方針

### 7.1 状態の最小化
このフェーズでは、以下の最小 state に絞る。

- `content`
- `uiState`: `idle | memo_selected | editing`
- `isVisible`
- `position`
- `size`
- `isDirty`

やらないこと:
- セッション選択
- 複数メモ
- セッション単位削除
- autosave
- DB永続化
- Delete キー / `delete_confirm` 状態

補足:
- Phase 2 では Delete キー入力は無視する
- `closed` は選択状態ではなく表示状態なので、`uiState` に含めず `isVisible` で扱う

理由:
- 1枚セッションの体験検証に不要
- 先に状態数を増やすと破綻しやすい

### 7.2 保存の扱い
- このフェーズの保存は `メモリ上の更新保存` でよい
- `Cmd + S` と `Cmd + Enter` の体験確認を先に行う
- DB保存は Phase 4 に回す
- dirty 判定は `isDirty` のみで持つ

### 7.3 リサイズの扱い
- サイズは React state で持つ
- 永続保存はしない
- 再表示時にデフォルトサイズへ戻る仕様と整合する

---

## 8. Gate 条件（Exit Criteria）
Phase 2 全体の完了条件は以下。

1. 1枚メモが sticky の見た目で表示される
2. シングルクリックで選択できる
3. ダブルクリックまたは `Enter` で編集できる
4. テキスト入力ができる
5. ドラッグ移動ができる
6. 角ドラッグでリサイズできる
7. `Cmd + S` で更新保存できる
8. `Cmd + Enter` で保存して閉じられる
9. 閉じた後にグローバルホットキーで再表示できる
10. 前面レイヤーとしての既存疎通を壊していない

## 9. 回帰 / 副作用チェック

### UI / UX
- メモ選択時に誤って編集開始しないか
- ダブルクリックでのみ編集開始するか
- `Esc` で編集終了できるか
- ドラッグ開始前に `6px` のしきい値があるか
- リサイズ中に選択状態が壊れないか

### 前面レイヤー
- `Cmd + Shift + 1` の表示/非表示が壊れていないか
- `Cmd + Shift + 2` の click-through 切替が壊れていないか
- 前面レイヤーが再び黒/白背景の巨大ウィンドウ化していないか

### 入力
- `Cmd + S` が通常文字入力と競合しないか
- `Cmd + Enter` が改行と競合しないか
- 日本語入力中にショートカットや Enter 起因の状態遷移が不自然に発火しないか

## 10. 既存実装との差分
- 現在は `overlay probe` 画面であり、本来のメモUIではない
- 現在はショートカット疎通のみ重視している
- 本計画では、`前面レイヤーの成立確認済み` を前提に、メモUIの最小体験へ進める

差分によるリスク:
- 疎通確認用UIを消しすぎると、デバッグしにくくなる
- ただしプローブを残しすぎると本実装と責務が混ざる

方針:
- デバッグ表示は最小限に縮小する
- 必要なら `dev` 時だけ見える形にする

---

## 11. DRY / KISS 評価

### DRY
- このフェーズでは `MemoCard` 的な見た目と操作を `App.tsx` 直下で持ってよい
- 早い段階で抽象化しすぎず、実際に必要になった時点で共通化する

### KISS
- まずは1枚セッション専用として実装する
- 複数メモや DB 永続化を先に混ぜない
- 将来必要な state を先回りで持たない
