# Sticky AI / バイブコーディング開発注意事項 v1.0

## 1. 目的
この文書は、`sticky` の AI 主導開発において、短絡的実装・仕様逸脱・状態破綻・UI事故を防ぐための開発ルールを定める。  
特に本プロダクトは、`最前面レイヤー`, `グローバルショートカット`, `セッション管理`, `選択/編集状態`, `自動保存` が絡むため、思いつき実装を禁止し、段階的に検証しながら進める。

## 2. 必ず参照する SSOT / 基本文書
実装・計画・レビュー時は、以下を単一の真実源として扱う。

- `要件定義.md`
- `事前検討まとめ.md`
- `画面一覧_状態遷移_DBスキーマ案.md`
- `操作一覧表.md`
- `画面ワイヤー仕様.md`
- `デザインシステム初版.md`
- `疎通確認項目と確認方法.md`
- `疎通確認結果.md`
- `ロードマップ.md`

ルール:
- 仕様判断に迷ったら、まず上記を確認する
- 実装が文書とズレる場合は、先に文書を更新するか、差分理由を明文化する
- 会話内の口頭合意より、更新済み文書を優先する

## 3. sticky の開発原則
- 最優先は `思考の初速` と `操作の自然さ`
- 高機能化より `早く出せる / すぐ書ける / すぐ閉じられる` を優先する
- クラウド同期・共有・AI整理などは初期MVPに入れない
- 本文はプレーンテキストで固定し、リッチ機能は後回し
- 新機能を足す前に、既存の状態モデルと保存モデルを壊さないか確認する
- 一般的な macOS の挙動から逸脱する場合は、理由を文書化する

## 4. Do / Don’t
### Do
- 実装前に SSOT と現在の関連文書を確認する
- 変更は `最小単位` で入れる
- 状態遷移・保存仕様・選択モデルに影響する変更は先に文章で固定する
- 疎通確認が必要な箇所は、いきなり本実装せずプローブを作る
- `Session / Memo / Slot / Selection` の責務分離を維持する
- 自動保存・自動閉じる・削除は必ず失敗系も考慮する
- 実装後は `build / typecheck / 疎通確認` を行う
- 文書間の食い違いを見つけたら、先に是正する

### Don’t
- 仕様未確定のまま UI だけ先に作り込まない
- Tauri / macOS の制約を無視して「後で何とかなる」と考えない
- 状態を複数箇所で二重管理しない
- ローカル保存ロジックを UI イベントごとにバラバラに書かない
- デバッグ用ショートカットや暫定UIを残したまま本実装に混ぜない
- 会話だけで合意して文書更新を省略しない
- 1フェーズで多目的実装をしない

## 5. sticky 特有の高リスク領域
このプロジェクトでは以下を高リスクとみなし、実装前に確認を要する。

1. `透過前面レイヤー`
- 他アプリの上に出る
- 必要時のみ入力を受ける
- 背面アプリ操作を阻害しない

2. `グローバルショートカット`
- macOS上で安定して動くか
- 権限・環境差異で壊れないか

3. `クリック / 選択 / 編集モデル`
- 誤選択や誤ドラッグが起きないか
- macOSユーザーにとって学習コストが高すぎないか

4. `保存モデル`
- autosave と明示保存の整合
- 空メモ / 未使用セッション / クラッシュ復帰時の cleanup

5. `セッション管理`
- session_id の更新で論理整合性が保たれるか
- 色スロットの解放・再利用が壊れないか

## 6. 計画書の必須構成
AI に計画を作らせる場合、最低限以下を含める。

1. `SSOT参照宣言`
2. `今回触る関連ファイル`
3. `問題一覧`
4. `修正フェーズ`
5. `Gate条件`
6. `回帰/副作用チェック`
7. `変更履歴`

## 7. 問題一覧の書き方
問題は必ず ID 付きで管理する。  
sticky では以下の体系を推奨する。

- `A-`: Architecture / 基盤・構造
- `S-`: State / 状態管理
- `U-`: UI / UX
- `D-`: Data / 保存・DB
- `T-`: Tauri / OS疎通
- `K-`: Knowledge / 文書・仕様整合

例:
- `T-01: 透明前面レイヤーのクリック透過が未検証`
- `S-03: セッション選択状態への導線が未確定`

## 8. MECE 検査
計画作成後、必ず以下を確認する。

### 検査A: Issue → Phase 対応
- すべての Issue に対応する Phase がある
- すべての Phase に対応する Issue がある

### 検査B: SSOT 整合
- 要件定義 / 状態遷移 / DB設計 / 操作一覧と矛盾していない
- 矛盾があるなら修正理由を書いている

### 検査C: DRY / KISS
- 重複状態や重複ロジックを増やしていない
- 複雑化している場合、その理由がある
- Tauri 制約への対応として必要以上の抽象化をしていない

## 9. フェーズ管理の閾値
AI 実装は小さく進める。

### 1フェーズの上限
- 触るファイル数: 最大 `3`
- 新規ファイル: 最大 `1`
- 主目的: `1つ` のみ

### 逸脱時
- 閾値を超える場合は、先に計画を分割する
- 「ついでにこれも」は禁止

## 10. sticky における保存・状態管理ルール
- 保存は `追加保存` ではなく `更新保存`
- autosave も明示保存と同じ保存経路を通す
- `title` は保存時に `content` 先頭10文字から再生成する
- 管理画面には `content != ''` のメモのみ表示する
- 空メモは DB に作ってもよいが、cleanup ルールを必ず持つ
- 起動時は `is_open` を全リセットする
- その後、空メモと空セッションを cleanup する

## 11. sticky における UI / 操作の原則
- 本番のメモUIはクロームレス
- 管理画面は通常の macOS ウィンドウ
- クリック・選択・編集の導線は必ず1箇所で定義する
- drag threshold は固定値を持つ
- 右クリックメニューの意味を曖昧にしない
- デスクトップUIのショートカットは増やしすぎない

## 12. 疎通確認の扱い
以下は実装前に独立確認を優先する。

- 透過前面レイヤー
- グローバルショートカット
- クリック透過切替
- SQLite保存
- 複数メモ描画
- リサイズ

ルール:
- 疎通確認は本実装とは分ける
- プローブコードは後で削除または置換する
- 疎通確認結果は文書に残す
- 疎通確認が通らない場合、本実装を進めない

## 13. よくある失敗と回避
### 失敗
- 会話だけで仕様変更し、文書が古いままになる
- UIだけ先に作り、状態管理が後追いで破綻する
- autosave と明示保存で別ロジックになる
- Tauri の制約を無視して React 側だけで成立すると考える
- セッション操作を直感だけで決めて、ドラッグ/選択が破綻する
- 暫定ログやプローブ用ショートカットを残す

### 回避
- 実装前に対象文書を更新する
- 保存経路を一本化する
- 高リスク項目は先に単体疎通確認する
- 実装後に build / cargo check / 実機操作を行う
- 不要な暫定物は削除するか、明確に TODO 化する

## 14. 計画違反の定義と処遇
### 違反
- SSOTに反する実装
- 計画外の仕様追加
- 状態管理の二重化
- 保存モデルの勝手な変更
- 高リスク領域を疎通確認なしで本実装する行為

### 処遇
1. 実装停止
2. 何がズレたか記録
3. 文書修正または計画作り直し
4. 必要なら例外扱いとして根拠を明文化

## 15. 例外適用ルール
以下の場合のみ、計画逸脱を許容する。

- 透明前面レイヤーが Tauri で成立しない
- macOS 権限問題でショートカット方式を変更せざるを得ない
- 本番に影響する重大な操作不能バグ
- SSOT 間に解決不能な矛盾

例外時の必須要件:
- 根拠を書く
- 暫定対応であることを明示する
- 正常復帰計画を書く
- ロールバック方針を書く

## 16. 提出前セルフチェック
```md
## セルフチェック結果

### SSOT整合
[ ] 要件定義を確認した
[ ] 状態遷移文書を確認した
[ ] 操作一覧を確認した
[ ] DB設計と矛盾していない
[ ] 事前検討まとめと矛盾していない

### 変更範囲
[ ] 1フェーズの主目的は1つ
[ ] 触るファイル数は3以下
[ ] 新規ファイルは1以下

### 状態・保存
[ ] 保存経路は一本化されている
[ ] autosave でも title 更新が漏れない
[ ] cleanup ルールを壊していない
[ ] is_open の扱いに矛盾がない

### UI / UX
[ ] クリックモデルに矛盾がない
[ ] drag threshold を壊していない
[ ] 暫定UIやデバッグ要素を残していない

### Tauri / 疎通
[ ] 高リスク変更なら疎通確認項目を更新した
[ ] build / cargo check / 必要な実機確認を行った

### 判定
全項目OKであれば実装またはレビュー依頼へ進む
```

## 17. 独立レビュー用プロンプト
```md
あなたは sticky の設計・実装レビュアーです。
作成者とは別視点で、以下の計画または差分をレビューしてください。

観点:
1. 要件定義・状態遷移・操作一覧・DB設計と矛盾がないか
2. Tauri / macOS の制約を無視していないか
3. 保存モデル、autosave、cleanup に破綻がないか
4. セッション管理と選択モデルに曖昧さがないか
5. MVPとして削るべきもの、足りないものは何か
6. DRY / KISS を壊していないか

出力形式:
- BLOCKER
- ADVISORY
- このまま進めてよい点
- 次の具体アクション

注意:
- 文書に明示されていないものは「欠落」として扱う
- 会話の意図を推測して補わない
```

## 18. この文書の目的
- sticky の AI 開発を短絡的な実装から守る
- 文書と実装の整合性を維持する
- 高リスクな Tauri / macOS 領域を先に検証する
- 保守性・再利用性・段階実装を優先する
